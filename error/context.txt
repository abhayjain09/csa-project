Here's how a single document request flows through the resolver tiers, start to finish, based on the orchestrator in `agent.py` (`_invoke_sync`).

## Before the tiers: query preparation
The payload arrives either as a structured `reports[]` list or legacy `web_query<N>` fields. For each report the agent builds a query, then `_prepare_query` runs: resolve relative years, clean markdown/brackets, and an LLM rewrite into one focused search string. `_apply_latest_completed_fiscal_year` injects the latest completed FY for recurring classes (annual report) when no year is given. A per-query budget (`_QueryBudget`) is created — a shared verify counter and wall-clock deadline threaded through every tier, so the whole chain stops when exhausted. The document class is matched (`_matched_doc_classes`), which drives aliases, validation contracts, and registry routing.

## Tier 1 — Web search (Vertex grounded, Gateway fallback)
The primary query is fanned out into variants: regional aliases (`_alias_queries`) plus LLM-generated variants (`_llm_generate_search_queries`) — site-scoped, unscoped, IR-subdomain, and registry-scoped forms. These run in parallel (`_parallel_web_search`). Each query hits the **Vertex Lambda** (`SEARCH_BACKEND=vertex_lambda`); if Vertex returns nothing and fallback is on, it drops to the **AgentCore Gateway** managed WebSearch. Results merge/dedupe by URL, then get filtered (junk hosts, sample/template, and — new — wrong-class filenames like DEFA14A), ranked, HEAD-pre-filtered, sampled, and passed to one grouped fail-closed LLM selection (`_llm_select_best` + `_confident`). If a confident doc-file is selected it's committed at `stage=search`; if the pick is an HTML landing page, `_resolve_from_html` digs the linked document out.

## Tier 2 — Official registry (EDGAR + Companies House)
If Tier 1 finds nothing, and the class has a registry mapping in `report_specs`, `registry_tier.registry_resolve` runs. EDGAR serves annual reports (10-K/20-F/40-F) and proxy statements (**DEF 14A only** now); Companies House serves UK annual accounts. Jurisdiction orders which registry is tried first. Deterministic hits (CIK + form-type match) skip the LLM verifier entirely; non-deterministic hits still pass the same fail-closed check. This tier is important because **it isn't WAF-blocked** — it's the off-site path for annual/proxy even when a company's own domain blocks fetches.

## Tier 3a — Sitemap enumeration
Still nothing → `_sitemap_resolve` pulls robots.txt/sitemaps for the domain, scores likely report/policy hub pages, fetches a bounded set of landing pages, extracts document links from them, and class-verifies each (most-recent-year preferred).

## Tier 3b — Deep static crawl
`_deep_static_crawl` recursively fetches same-domain HTML from the site root, regex-harvests document links per page, and verifies each. For filing classes (annual/proxy/remuneration) the **browser tier is tried before this** (browser-first), since JS-heavy IR pages resolve better and cheaper.

## Tier 4 — AgentCore Browser (Playwright)
Last resort for JS-heavy sites: navigates the root, dismisses cookie modals, harvests links (including hover-revealed menus), fetches candidates in-session, falls back to click-to-download, then IR-prioritized deep navigation, then optional vision-guided click. Detects WAF/bot-challenge pages (`block_marker=access denied`) and skips clicking when blocked — which is exactly where S&P dies.

## Termination
The first tier to produce a **class-verified** document commits it (`_store`: content-hash S3 key, `.metadata.json` sidecar with `doc_class`, provenance row). If every tier fails or the budget is exhausted, it returns `NO DOCUMENT FOUND (failed closed)` — no guessing.

Every tier shares one verifier (`_llm_select_best`/`_confident`) and one budget, so the chain is fail-closed and bounded end to end. The one thing this flow can't survive is a fetch-layer block (the S&P WAF): search finds the doc, but every tier that must *fetch* it 403s, so it correctly ends in a closed miss.
